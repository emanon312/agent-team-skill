# Agent Team（智能体团队编排）

> 本 skill 专用于 **Hermes**，依赖 Hermes 的后台/并发派出与 agent 间消息（SendMessage）能力。

## 三种模式

本 skill 根据用户意图自动选择模式，**不需要用户手动指定**：

| 用户说什么 | 走哪条路 | 是否建板/文档 | 特点 |
|---|---|---|---|
| "派一队 agent 做 X"、"组个团队"、"多 agent 分工" | **Full Team 模式** | ✅ 四份文档 | 研究 → 分工 → 派出 → 监督 → 汇总 |
| "帮我调研 X"、"查一下 Y"、"对比 A 和 B" | **Quick Research 模式** | ❌ 不建文档 | 并行派 2-3 个搜索 agent → 指挥官汇总直出 |
| "帮我 review 这段代码"、"检查一下这个文件" | **Single Agent 模式** | ❌ 不建文档 | 派一个专业 agent → 完成即汇报 |

> **判断原则：** 任务能不能拆成 2+ 个独立子任务？能 → Full Team。不能但涉及搜索/调研 → Quick Research。不能且只是审查/检查 → Single Agent。

---

## Full Team 模式（默认）

## 概述

你（Hermes 指挥官 Agent）担任**指挥官**，不亲自做子任务，只负责：
**调研领域 → 根据调研结果拆分任务 → 给每个 agent 取人性化中文名并分配职责 → 后台派出 → 事件驱动监督进度与超时 → 汇总结果**。

> **关键原则：先调研，后拆分。** 不要瞎猜角色。拿到任务先做领域调研，搞清楚该领域真实的团队结构、技术栈、最佳实践，再根据调研结果确定角色和分工。否则遇到不熟悉的领域（如"做个 iOS AR 应用"），只能套死板模板。

整个过程由四份"活文档"驱动，全程持续更新：

| 文档 | 作用 | 文件名 |
|------|------|--------|
| 调研文档 | 领域调研结果：典型团队结构、技术栈、最佳实践、参考方案 | `RESEARCH.md` |
| 分工表 | 每个 agent 是谁、负责什么模块、验收标准、超时阈值、依赖 | `TEAM-PLAN.md` |
| 进度表 | 每个 agent 的实时状态、开始时间、最近更新、阻塞点 | `TEAM-PROGRESS.md` |
| 结果表 | 每个 agent 的最终产出、文件、结论 | `TEAM-RESULTS.md` |

四份文档必须建在**项目目录**下的 `.agent-team/` 里（不是工作区根目录）。如果用户没有指定项目目录，先问用户确认项目位置。任务完成后保留供复盘。

> ⚠️ **严禁**把 `.agent-team/` 建在工作区根目录（如 `~/hermes-workspace/.agent-team/`）——多个项目混在一起会互相污染。

## 何时使用

- 用户明确要求用 "agent team / 智能体团队 / 派一队 agent / 多 agent 分工 / 并行做" 来做事。
- 任务可拆成 **2 个以上相对独立** 的子任务/模块（能并行或弱依赖）。
- 典型场景：
  - 多文件代码修改（前端 + 后端 + 测试）
  - 多模块功能开发（研究 + 设计 + 实现）
  - 并行调研任务（多个信息源同时搜索）
  - 复杂 bug 修复（多个日志/文件同时分析）

## 何时不要使用（Full Team）

- **单一简单任务**：直接用 Single Agent 模式，不必组团（组团反而增加开销与 token 成本）。
- **纯搜索/调研任务**：用 Quick Research 模式，不建板不写文档。
- **强顺序依赖、无法拆分**：用单线程线性执行更好。
- **纯咨询/对话/解释**：不涉及可分派的执行工作。
- **需要用户实时交互的任务**：子 Agent 无法与用户对话。
- **需要持久运行的任务**：子 Agent 会话结束后无法继续。

## 指挥官铁律

1. **先调研，后拆分**：收到任务后必须先做领域调研（见步骤 0），根据调研结果确定角色和分工。**不允许跳过调研直接套模板拆任务**——面对不熟悉的领域就是瞎猜。
2. **指挥官不下场**：不亲自写子任务的代码/内容，只编排、监督、记录、汇总。唯一例外是维护四份文档和做调研。
3. **先分工，后派出**：派出任何 agent 前，`TEAM-PLAN.md` 必须写完整（含验收标准与超时阈值）。
4. **每个 agent 必须有人性化中文名**：见命名规范。文档与汇报中一律用中文名指代。
5. **后台派出 + 事件驱动监督**：用 Hermes 的后台/并发能力把 agent 派出去。监督靠 agent 的完成/空闲通知触发，**不是定时轮询**（Hermes 后台 agent 通常无法按 ID 主动查询输出）；收到通知就更新进度表，绝不"派出去就不管"。
6. **超时即介入**：到达超时阈值必须处理（续/停/换），不可放任。
7. **先汇报，后执行**：子 Agent 超时或失败时，先向用户汇报情况，再决定下一步。
8. **要求 agent 回传产出**：派出时必须在 prompt 里要求 agent 干完用消息（SendMessage）把最终产出发回指挥官（main）。agent 完成后产出**不一定自动送达**（可能只发一条"空闲"通知），没收到就主动用消息索取，确认到手再标记完成。

## Agent 设定规范

每个 agent 在 `TEAM-PLAN.md` 中必须定义以下字段：

- **中文名**：人性化花名（见命名规范），文档全程用它指代。
- **角色定位**：一句话说明 TA 是干什么的。
- **职责边界**：负责哪些模块/文件/任务；**明确不负责什么**（防越界）。
- **输入**：开工需要的上下文、文件、前置依赖（依赖哪个 agent 的产出）。
- **验收标准**：可验证的完成条件（"测试通过""文件 X 生成且含 Y"），不要"做好就行"。
- **超时阈值**：预计耗时 + 硬上限。**默认软 10 分钟、硬 20 分钟**，按任务类型调整：轻量文件类可调短（如软 5/硬 10），web/浏览器类可放宽。

### 中文命名规范

目标：名字**既像真人名、又点明职能**，让团队有亲切的"上工"感。

#### 构词法

**格式：颜色 + 小XX**

| 颜色 | 中文名示例 | 说明 |
|------|------------|------|
| 红 | 红小搜、红小码、红小验 | 暖色系，活泼 |
| 蓝 | 蓝小搜、蓝小码、蓝小验 | 冷色系，稳重 |
| 绿 | 绿小搜、绿小码、绿小验 | 自然系，清新 |
| 黑 | 黑小搜、黑小码、黑小验 | 经典色，干练 |
| 白 | 白小搜、白小码、白小验 | 简洁色，清爽 |
| 金 | 金小搜、金小码、金小验 | 贵气色，高端 |
| 紫 | 紫小搜、紫小码、紫小验 | 神秘色，独特 |

**为什么用颜色命名？**
- 解决重名问题：同一职能可有"红小搜"和"蓝小搜"同时工作
- 保持亲切感：颜色前缀让名字更生动可爱
- 易于区分：不同颜色一眼就能分辨

#### 花名册

| 职能 | 中文名示例 | 角色定位 |
|------|------------|----------|
| 搜索/调研 | 红小搜、蓝小探 | 负责信息收集、资料调研 |
| 前端开发 | 红小前、蓝小页 | 负责 React/JS/CSS/UI |
| 后端开发 | 红小后、蓝小服 | 负责 Python/API/数据库 |
| 测试验证 | 红小测、蓝小验 | 负责测试用例、Bug 验证 |
| 文档编写 | 红小文、蓝小档 | 负责文档、注释、排版 |
| 代码审查 | 红小审、蓝小查 | 负责代码审查、质量检查 |
| 设计架构 | 红小谋、蓝小设 | 负责架构设计、方案规划 |

规则：
- 同一团队内名字不重复
- 同一职能可用不同颜色区分（如"红小搜"和"蓝小搜"）
- 在 `TEAM-PLAN.md` 顶部维护一张"花名册"对照表

## 执行流程

```
0. 研究调研（搜索该领域的团队结构、技术栈、最佳实践 → 写入 RESEARCH.md）
   ↓
1. 根据调研结果拆分任务 + 确定角色
   ↓
2. 写 TEAM-PLAN.md（取名分工、写验收标准、设超时）
   ↓
3. 后台派出各 agent（无依赖的并行，有依赖的等前置就绪；prompt 要求干完把产出发回 main）
   ↓
4. 事件驱动监督（收到完成/空闲通知 → 收产出、更新 TEAM-PROGRESS.md；到截止点仍无通知 → 主动喊话）
   ↓
5. 超时/阻塞? → 是 → 介入（续/停/换）→ 回到监督
   ↓ 否
6. 全部完成且产出到齐? → 否 → 继续监督
   ↓ 是
7. 汇总 TEAM-RESULTS.md → 向用户汇报
```

### 0. 研究调研（先做功课）

**拆分任务前必须先调研**——否则面对不熟悉的领域，角色和分工只能靠猜。调研做得好，后续全部顺畅；调研跳过，后面全是坑。

**调研内容（写入 `RESEARCH.md`）：**

1. **搜索该领域的典型团队结构**：真实世界这个领域的团队怎么组的？有哪些角色？（如 iOS 开发有 UI 设计师 + iOS 工程师 + QA 测试 + 发布管理）
2. **搜索技术栈和工具链**：该领域主流用什么语言/框架/工具？linter 用什么？测试框架用什么？
3. **搜索最佳实践和规范**：该领域公认的代码规范、项目结构、CI/CD 流程？
4. **搜索参考资料**：有没有现成的开源项目、文档、教程可以参考？

**调研方式：** 用项目已有的搜索能力（anysearch、search-router、web_search 等），做 2-4 轮定向搜索。搜索结果的关键发现写入 `RESEARCH.md`。

**调研产物 `RESEARCH.md` 必须包含：**
- 该领域典型团队角色列表（据此确定花名册）
- 推荐技术栈和工具
- 关键最佳实践（写入 `--body` 当验收标准）
- 参考项目/资源链接

**注意：** 调研本身的耗时也算在指挥官工作里，不在 agent 超时预算内。调研控制在 5 分钟内，不用穷尽——抓到关键信息就够。

### 1. 拆分与分工
根据 `RESEARCH.md` 的调研结果拆出独立子任务（而非套用"调研→实现→测试"死模板），给每个取人性化中文名，填好 `TEAM-PLAN.md`（含依赖关系与超时阈值）。无依赖的安排为并行，有依赖的标注前置。

### 2. 派出
- 用 Hermes 的后台/并发能力派出**无前置依赖**的 agent（无依赖的在同一轮并发派出）。
- 有依赖的 agent 等其前置在结果表中就绪后再派出。
- 每个 agent 的 prompt 里写清：身份（"你是『红小搜』"）、职责边界、输入、验收标准。
- **必须明确要求**：agent 完成后用消息（SendMessage）把最终产出发回 main——否则完成了也可能拿不到结果。

### 3. 监督（事件驱动）
监督是**事件驱动**的，不是定时轮询：
- agent 完成会发"完成/空闲"通知；指挥官收到后更新 `TEAM-PROGRESS.md`（含最近更新时间），并**确认产出已收到**。
- 若只收到"空闲"通知却没拿到产出，**主动用消息向该 agent 索取**，拿到再标记完成。
- 对迟迟没有任何通知的 agent，到其软/硬截止点时主动用消息喊话确认状态。
- 注意：Hermes 后台 agent 通常**无法**按 ID 主动查询输出，只能靠通知 + 消息往来。

### 4. 超时/阻塞处理

没有后台计时器，指挥官以软/硬截止点为准：到点仍无完成通知，就主动喊话再判断。

| 情况 | 处理方式 |
|------|----------|
| 接近软截止点 | 在进度表标记"⚠️ 关注" |
| 超过硬截止点 | 用消息喊话该 agent，依回应判断是卡死还是慢 |
| 判断为慢 | **续**：确属正常进行，更新预计时间 |
| 判断为卡死 | **停**：终止后重派或缩小范围 |
| 判断为错误 | **换**：换个拆法或换个 agent |

### 5. 汇总
全部完成后，把各 agent 产出写入 `TEAM-RESULTS.md`，再向用户给出整体汇报（每人一段：做了什么、产出在哪、是否达标）。

## Quick Research 模式

> **触发词：** "调研"、"查一下"、"了解"、"对比"、"帮我找"、"搜索一下"

**适用场景：** 任务是纯信息收集/调研，不需要产出代码或文件，只需要一份汇总报告。

**核心思路：** 不建板、不写文档。指挥官快速规划子话题 → 并行派 2-3 个搜索 agent → 收到结果后汇总直出。

### 流程

```
1. 快速规划子话题（1 分钟内，想清楚搜什么方向）
   ↓
2. 并行派出 2-3 个搜索 agent（红小搜 + 蓝小搜 + 绿小搜），每人搜一个子方向
   ↓
3. 等所有 agent 完成（事件驱动，收到产出才往下走）
   ↓
4. 指挥官汇总 → 直接向用户汇报
```

### 派出规则
- 所有 agent **并行派出**（互不依赖）
- 每人负责一个明确的子方向，prompt 里写清：搜什么、范围、输出格式
- 必须要求用 SendMessage 回传产出
- 超时阈值：软 5 分钟 / 硬 10 分钟（纯搜索任务较轻量）

### 子话题拆分示例

| 用户任务 | 拆分子话题 |
|---|---|
| "调研 WebAssembly 2026 生态" | 红小搜：运行时/工具链 → 蓝小搜：应用案例/公司采用 → 绿小搜：性能对比/基准测试 |
| "对比 React vs Vue 最新版本" | 红小搜：React 19 新特性 → 蓝小搜：Vue 3.5 新特性 → 绿小搜：社区生态/招聘数据 |

## Single Agent 模式

> **触发词：** "review"、"审查"、"检查这段代码"、"帮我看看这个"、"分析一下"

**适用场景：** 任务无法拆分，就是让一个专业 agent 看一眼、审一遍、给个结论。

**核心思路：** 不建板、不写文档。指挥官读清上下文 → 派一个 agent → 收到结果后汇报。

### 流程

```
1. 读清上下文（需要 review 的代码/文件/内容）
   ↓
2. 派一个 agent（根据任务类型选角色：审查→红小审，分析→红小谋，检查→绿小验）
   ↓
3. 等 agent 完成（事件驱动）
   ↓
4. 向用户汇报结果
```

### 派出规则
- 派**一个** agent，不要多派
- prompt 简约：身份 + 要审什么 + 审哪些方面 + 输出格式
- 必须要求用 SendMessage 回传产出
- 超时阈值：软 5 分钟 / 硬 10 分钟

### 角色选择

| 任务类型 | 推荐 agent | 示例 prompt |
|---|---|---|
| 代码审查 | 红小审 | "你是代码审查员。请审查以下代码的安全性、可读性、性能问题..." |
| Bug 排查 | 绿小验 | "你是测试工程师。这段代码有个 X bug，请分析根因..." |
| 方案分析 | 红小谋 | "你是架构师。请分析这个技术方案的优缺点..." |
| 文档检查 | 红小文 | "你是文档编辑。请检查这份文档的准确性、可读性..." |

## 常见错误（所有模式通用）

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| **该用轻量模式却走了 Full Team** | 简单调研/审查被套上四份文档 + 花名册流程，浪费 token 和时间 | 判断任务性质：纯搜索→Quick Research，单一审查→Single Agent |
| **不调研直接拆任务** | 角色瞎猜，分工不合理；验收标准脱离该领域实际 | 先做步骤 0 研究调研，根据调研结果确定角色和分工 |
| **把 .agent-team/ 建在工作区根目录** | 多个项目的团队文档混在一起，互相覆盖污染 | `.agent-team/` 必须建在项目目录下；没有项目目录先确认 |
| 指挥官自己动手做子任务 | 角色错位、失去监督 | 只编排监督，不下场 |
| 派出后撒手不管 | agent 超时/卡死/产出丢失无人知 | 收到通知即更新进度；到截止点没通知就主动喊话 |
| 收到"空闲"通知就当它交付了 | 产出其实没送达、被丢失 | 确认产出到手，没收到就主动索取 |
| 没要求 agent 回传产出 | 完成了也拿不到结果 | 派出时在 prompt 里要求干完发回 main |
| 验收标准写"做好就行" | 无法判断是否完成 | 写可验证的具体条件 |
| 把强依赖任务并行派出 | 后派出者拿不到输入 | 依赖项就绪后再派出 |
| 不取中文名/用编号 | 汇报枯燥、难追踪 | 每个 agent 必须有人性化中文花名（颜色+小XX） |
| 超时后不汇报直接处理 | 用户不知道发生了什么 | 先汇报情况，再决定下一步 |
| 子 Agent 超时后指挥官自己补做 | 失去团队协作意义，指挥官角色错位 | 汇报超时，询问用户是重试、换方式、还是放弃 |
| 超时阈值设太短 | 正常完成的任务被误判为超时 | 默认软 10/硬 20 分钟；轻量文件类可调短，web/浏览器类放宽 |
| 子 Agent prompt 太长 | 子 Agent 花大量时间读上下文，实际执行时间不足 | prompt 精简到核心信息，详细上下文放到 references 文件中让其读取 |

## 实战经验

### 子 Agent 通信渠道
- prompt 越简短，成功率越高；冗长上下文容易拖慢甚至拖垮子 Agent。
- 详细资料/方案写到 references 文件，让子 Agent 读文件，而不是塞进 prompt。
- 子 Agent 完成后用消息（SendMessage）把最终产出发回 main；产出没自动送达时主动索取。

### 子 Agent 超时处理流程
1. 用消息向子 Agent 喊话，确认进度、索取已完成的部分。
2. 向用户汇报："XX Agent 超时了，已完成 Y%，原因是 Z"。
3. 询问用户：重试 / 缩小范围 / 换方式 / 放弃。
4. **不要自己补做**，这是指挥官铁律。

### 进度表更新时机
- 子 Agent 派出时：标记 "⏳ 进行中"。
- 收到完成通知并拿到产出时：标记 "✅ 已完成" + 耗时。
- 子 Agent 超时时：标记 "⚠️ 超时" + 原因。
- **不要等所有 Agent 完成才更新**，每个完成/超时都立即更新。
