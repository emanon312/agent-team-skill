# Agent Team（智能体团队编排）

## 概述

你（Hermes 指挥官 Agent）担任**指挥官**，不亲自做子任务，只负责：
**拆分任务 → 给每个 agent 取人性化中文名并分配职责 → 后台派出 → 事件驱动监督进度与超时 → 汇总结果**。

整个过程由三份"活文档"驱动，全程持续更新：

| 文档 | 作用 | 文件名 |
|------|------|--------|
| 分工表 | 每个 agent 是谁、负责什么模块、验收标准、超时阈值、依赖 | `TEAM-PLAN.md` |
| 进度表 | 每个 agent 的实时状态、开始时间、最近更新、阻塞点 | `TEAM-PROGRESS.md` |
| 结果表 | 每个 agent 的最终产出、文件、结论 | `TEAM-RESULTS.md` |

三份文档默认建在工作目录下的 `.agent-team/` 目录里（任务完成后保留供复盘）。

## 何时使用

- 用户明确要求用 "agent team / 智能体团队 / 派一队 agent / 多 agent 分工 / 并行做" 来做事。
- 任务可拆成 **2 个以上相对独立** 的子任务/模块（能并行或弱依赖）。
- 典型场景：
  - 多文件代码修改（前端 + 后端 + 测试）
  - 多模块功能开发（研究 + 设计 + 实现）
  - 并行调研任务（多个信息源同时搜索）
  - 复杂 bug 修复（多个日志/文件同时分析）

## 何时不要使用

- **单一简单任务**：直接做，或只派一个 agent，不必组团（组团反而增加开销与 token 成本）。
- **强顺序依赖、无法拆分**：用单线程线性执行更好。
- **纯咨询/对话/解释**：不涉及可分派的执行工作。
- **需要用户实时交互的任务**：子 Agent 无法与用户对话。
- **需要持久运行的任务**：子 Agent 会话结束后无法继续。

## 指挥官铁律

1. **指挥官不下场**：不亲自写子任务的代码/内容，只编排、监督、记录、汇总。唯一例外是维护三份文档。
2. **先分工，后派出**：派出任何 agent 前，`TEAM-PLAN.md` 必须写完整（含验收标准与超时阈值）。
3. **每个 agent 必须有人性化中文名**：见命名规范。文档与汇报中一律用中文名指代。
4. **后台派出 + 事件驱动监督**：用 Hermes 的后台/并发能力把 agent 派出去。监督靠 agent 的完成/空闲通知触发，**不是定时轮询**（后台 agent 通常无法按 ID 主动查询输出）；收到通知就更新进度表，绝不"派出去就不管"。
5. **超时即介入**：到达超时阈值必须处理（续/停/换），不可放任。
6. **先汇报，后执行**：子 Agent 超时或失败时，先向用户汇报情况，再决定下一步。
7. **要求 agent 回传产出**：派出时必须在 prompt 里要求 agent 干完用消息把最终产出发回指挥官（main）。agent 完成后产出**不一定自动送达**（可能只发一条"空闲"通知），没收到就主动用消息索取，确认到手再标记完成。

## Agent 设定规范

每个 agent 在 `TEAM-PLAN.md` 中必须定义以下字段：

- **中文名**：人性化花名（见命名规范），文档全程用它指代。
- **角色定位**：一句话说明 TA 是干什么的。
- **职责边界**：负责哪些模块/文件/任务；**明确不负责什么**（防越界）。
- **输入**：开工需要的上下文、文件、前置依赖（依赖哪个 agent 的产出）。
- **验收标准**：可验证的完成条件（"测试通过""文件 X 生成且含 Y"），不要"做好就行"。
- **超时阈值**：预计耗时 + 硬上限（默认软 10 分钟、硬 20 分钟，按任务调整）。

### 中文命名规范

目标：名字**既像真人名、又点明职能**，让团队有亲切的"上工"感。

#### 构词法

| 方式 | 格式 | 示例 |
|------|------|------|
| 小 + 职能字 | 小 + 单字 | 小搜、小码、小验、小文 |
| 小 + 职能词 | 小 + 双字 | 小搜查、小测试、小排版 |
| 姓 + 职能词 | 姓 + 双字 | 张搜查、李代码、王测试、周文档 |

#### 花名册

| 职能 | 中文名示例 | 角色定位 |
|------|------------|----------|
| 搜索/调研 | 小搜、小探、张搜查 | 负责信息收集、资料调研 |
| 前端开发 | 小前、小页、李前端 | 负责 React/JS/CSS/UI |
| 后端开发 | 小后、小服、王后端 | 负责 Python/API/数据库 |
| 测试验证 | 小测、小验、赵测试 | 负责测试用例、Bug 验证 |
| 文档编写 | 小文、小档、周文档 | 负责文档、注释、排版 |
| 代码审查 | 小审、小查、陈审查 | 负责代码审查、质量检查 |
| 设计架构 | 小谋、小设、吴设计 | 负责架构设计、方案规划 |

规则：同一团队内名字不重复；在 `TEAM-PLAN.md` 顶部维护一张"花名册"对照表。

## 执行流程

```
1. 拆分任务
   ↓
2. 写 TEAM-PLAN.md（取名分工、写验收标准、设超时）
   ↓
3. 后台派出各 agent（无依赖的并行，有依赖的等前置就绪；prompt 要求干完把产出发回 main）
   ↓
4. 事件驱动监督（收到完成/空闲通知 → 收产出、更新 TEAM-PROGRESS.md；到截止点仍无通知 → 主动喊话）
   ↓
5. 超时/阻塞? → 是 → 介入（续/停/换）→ 回到监督
   ↓ 否
6. 全部完成且产出到齐? → 否 → 继续监督
   ↓ 是
7. 汇总 TEAM-RESULTS.md → 向用户汇报
```

### 1. 拆分与分工
拆出独立子任务，给每个取人性化中文名，填好 `TEAM-PLAN.md`（含依赖关系与超时阈值）。无依赖的安排为并行，有依赖的标注前置。

### 2. 派出
- 用 Hermes 的后台/并发能力派出**无前置依赖**的 agent（无依赖的在同一轮并发派出）。
- 有依赖的 agent 等其前置在结果表中就绪后再派出。
- 每个 agent 的 prompt 里写清：身份（"你是『小搜查』"）、职责边界、输入、验收标准。
- **必须明确要求**：agent 完成后用消息（如 SendMessage）把最终产出发回 main——否则完成了也可能拿不到结果。

### 3. 监督（事件驱动）
监督是**事件驱动**的，不是定时轮询：
- agent 完成会发"完成/空闲"通知；指挥官收到后更新 `TEAM-PROGRESS.md`（含最近更新时间），并**确认产出已收到**。
- 若只收到"空闲"通知却没拿到产出，**主动用消息向该 agent 索取**，拿到再标记完成。
- 对迟迟没有任何通知的 agent，到其软/硬截止点时主动用消息喊话确认状态。
- 注意：后台 agent 通常**无法**按 ID 主动查询输出（如 TaskList/TaskOutput 会查不到），只能靠通知 + 消息往来。

### 4. 超时/阻塞处理

没有后台计时器，指挥官以软/硬截止点为准：到点仍无完成通知，就主动喊话再判断。

| 情况 | 处理方式 |
|------|----------|
| 接近软截止点 | 在进度表标记"⚠️ 关注" |
| 超过硬截止点 | 用消息喊话该 agent，依回应判断是卡死还是慢 |
| 判断为慢 | **续**：确属正常进行，更新预计时间 |
| 判断为卡死 | **停**：终止后重派或缩小范围 |
| 判断为错误 | **换**：换个拆法或换个 agent |

### 5. 汇总
全部完成后，把各 agent 产出写入 `TEAM-RESULTS.md`，再向用户给出整体汇报（每人一段：做了什么、产出在哪、是否达标）。

## 三份文档模板

### TEAM-PLAN.md（分工表）
```markdown
# 任务分工表

## 总目标
<一句话描述整体目标>

## 花名册
| 中文名 | 角色 | 负责模块 | 超时(软/硬) | 依赖 |
|--------|------|----------|-------------|------|
| 小搜 | 调研 | 网络搜索 | 5/10min | 无 |
| 小码 | 实现 | src/x | 10/20min | 小搜 |
| 小验 | 测试 | tests/ | 5/10min | 小码 |

## 各 Agent 详情

### 小搜（调研）
- **角色定位**：负责信息收集和资料调研
- **职责边界**：搜索相关文档、找到最佳实践
  - 不负责：代码实现、测试
- **输入**：任务描述、关键词
- **验收标准**：找到至少 3 个相关方案，整理成对比文档
- **超时阈值**：软 5 分钟 / 硬 10 分钟
```

### TEAM-PROGRESS.md（进度表）
```markdown
# 进度表（最近更新：14:09）

| 中文名 | 状态 | 开始 | 最近更新 | 进度 | 阻塞/备注 |
|--------|------|------|----------|------|-----------|
| 小搜 | ✅ 已完成 | 14:02 | 14:08 | 100% | 无 |
| 小码 | ⏳ 进行中 | 14:09 | 14:09 | 30% | 无 |
| 小验 | ⏸️ 待启动 | - | - | 0% | 等待小码完成 |

**状态取值**：
- ⏸️ 待启动
- ⏳ 进行中
- ✅ 已完成
- ⚠️ 超时（关注）
- ❌ 失败

## 时间线
- 14:02 任务开始，小搜派出
- 14:08 小搜完成调研
- 14:09 小码派出，开始实现
```

### TEAM-RESULTS.md（结果表）
```markdown
# 结果汇总

## 项目：<项目名>
## 完成时间：<时间>

### 小搜（调研）
- **产出**：`research/comparison.md`
- **结论**：找到 5 个方案，推荐方案 B（性能最好）
- **遗留**：无

### 小码（实现）
- **产出**：`src/feature.py`、`src/feature.test.py`
- **结论**：功能实现完成，测试通过
- **遗留**：需要优化性能

### 小验（测试）
- **产出**：`tests/test_feature.py`
- **结论**：15 个测试用例，全部通过
- **遗留**：无

## 汇总结论
整体完成情况良好。功能已实现并通过测试，建议后续优化性能。
```

## 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 指挥官自己动手做子任务 | 角色错位、失去监督 | 只编排监督，不下场 |
| 派出后撒手不管 | agent 超时/卡死/产出丢失无人知 | 收到通知即更新进度；到截止点没通知就主动喊话 |
| 验收标准写"做好就行" | 无法判断是否完成 | 写可验证的具体条件 |
| 把强依赖任务并行派出 | 后派出者拿不到输入 | 依赖项就绪后再派出 |
| 不取中文名/用编号 | 汇报枯燥、难追踪 | 每个 agent 必须有人性化中文花名 |
| 超时后不汇报直接处理 | 用户不知道发生了什么 | 先汇报情况，再决定下一步 |
| 收到"空闲"通知就当它交付了 | 产出其实没送达、被丢失 | 确认产出到手，没收到就主动索取 |
| 没要求 agent 回传产出 | 完成了也拿不到结果 | 派出时在 prompt 里要求干完发回 main |
